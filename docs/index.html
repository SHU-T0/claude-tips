field notes / finding your unknowns

間違いは、モデルではなく
あなたの「不明点」から出る

Claude 公式の記事 A field guide to Claude Fable: finding your unknowns の技術解説。あわせて、この方法論をそのまま Claude Code で回せるようにした finding-unknowns プラグインの使い方をまとめた。

原典 · claude.com/blog/a-field-guide-to-claude-fable-finding-your-unknowns
実装 · github.com/SHU-T0/claude-tips

Claude に機能を1つ頼む。90分後、自信たっぷりに間違ったものが返ってくる。モデルが馬鹿なわけではない。埋めるべき空白を、あなたが渡さなかっただけだ。Claude はその空白を「たぶんこうだろう」で埋め、その推測が外れた。

この空白のことを、元記事は unknowns（不明点） と呼ぶ。そして主張はシンプルだ。モデルが賢くなるほど、作業の質を決めるのはモデルの能力ではなく、あなたが自分の unknowns をどれだけ明確にできるかになる。だからこの記事は、その不明点を実装の前・最中・後に、安く見つけるための手順書になっている。

01地図と、実際の土地

手元の地図と、実際の土地は違う。Claude に渡すプロンプト・計画・コンテキストが地図で、実際のコードベース・ドメインの制約・あなたが本当に望んでいるものが土地だ。両者がずれている場所、それが unknowns。Claude がそこに踏み込むと、意図を推測するしかなくなる。任せる範囲が広いほど、踏み込む空白も増える。

厄介なのは、このずれを直すコストが時間とともに跳ね上がる点だ。コードを書く前に気づけば、数分で直る。3つの PR を積んだ後に気づけば、その3つを巻き戻すことになる。仕様のわずかな変更が実装を大きく変えることは珍しくないし、そうなってからの revert は高い。やることは1つに絞られる。高くつく前に、知らなかったことを見つける。

02なぜ、いま不明点の話なのか

モデルが賢くなるほど、ボトルネックはモデル側からあなた側へ移る。能力で殴れる範囲が広がると、残るのは「何を作ればいいか」の解像度だ。だから診断のヒューリスティックはこうなる。長時間タスクの結果が間違って返ってきたら、プロンプトの語尾をいじる前に、自分が unknowns を定義しきれていないことを疑う。多くの場合、直すべきはそこだ。

補足しておく。これは Fable 専用の話ではない。ここで出てくる技法（盲点洗い、インタビュー、プロトタイプ、クイズ）は、特定のモデルの機能ではなく協働のやり方だ。Sonnet でも Opus でも同じだけ効く。Fable では unknowns が最大のレバーになる、という順序の問題にすぎない。

03不明点を4つに割る

不明点は一枚岩ではない。「自分が認識しているか」と「実際に把握できているか」の2軸で割ると、それぞれ打ち手が変わる。いちばん厄介なのは右下、存在にすら気づいていない領域だ。

ここで手を貸せるのが Claude 自身だ。コードもネットも速く読むし、失敗からの立て直しも速い。効かせるコツは1つ、自分の立ち位置を先に渡すこと。このコードベースにどれくらい慣れているか、このドメインをどれくらい知っているかを開示する。それだけで Claude の振る舞いが変わる。相手を「実行係」ではなく「相談相手」として使うということだ。

04前・最中・後で、不明点を狩る

実装の前

いきなり書かせない。まず盲点洗いをやる。「blind spot pass」「unknown unknowns」と言葉にして、考慮から漏れているものを挙げさせ、それを説明させる。ここが肝で、盲点は列挙されても意味が薄い。Claude がその背景を噛み砕いて教えてくれて初めて、次のプロンプトを知識で書ける。

インタビューは1問ずつ。しかも塗装の色ではなく、アーキテクチャを変える質問から。「重複行はスキップか上書きか」は後の全実装に響くが、「ボタンの角丸は何 px か」はいつでも直せる。各質問に推奨案を添えさせると、あなたは一言で採否できる。

言葉にしにくい好み（unknown knowns）は、説明を求めるより反応させるほうが早い。デザインなら、1案の色違いではなく、方向性からして違う3〜4案を捨て前提の HTML で出させ、どれが刺さるか反応する。プロトタイプにはもう1つ役がある。技術的にそもそも可能かを試す捨てスパイクだ。バックエンドを配線せず、動くかどうかだけ見る。

ただし、反応が効くのは「見れば良し悪しが分かる」ときだけだ。見ても判定できない領域では、選択肢を並べても選べない。元記事のいちばん鋭い教訓がここにある。

最初は何案か出して選ぶつもりだった。だが自分が「良い」が何か分かっていないと気づいた。だから選ぶのをやめ、Claude にカラーグレーディングそのものを教えてもらった。 — Fable 5 ローンチ動画を Claude Code だけで作った話より（要約）

選ぶ前に、教わる。判定基準を持ってから選択肢に向かう。参照を渡すときも同じ発想で、いちばん richな参照はスクショや言葉ではなくソースコードだ。別言語でも構わない。「このフォルダの、この部分を見て」と示すのがいちばん速い。

最後に計画。並び順に意味を持たせる。変わりやすいもの（データモデル・型の境界・UX フロー）を先頭に置く。ここがずれると波及が大きい。機械的なリファクタは末尾でいい。

実装の最中

どれだけ計画しても、土地は必ず何か返してくる。計画になかった判断を迫られた場所こそ、リアルタイムに現れた不明点だ。implementation-notes.md に3行で残す。Plan（計画は何と言っていたか）、Did（実際に何をしたか）、Why（土地が何を教えたか）。進捗ログではない。判断と驚きだけを書く。「インポータを実装した」は要らない。「アーカイブ済み行に API が null を返すので、インポータはそれを飛ばす」が要る。判断に迷ったら保守的な方を既定にして、記録して先に進む。可逆な選択でパスを止めない。

実装の後

2つの成果物で、信頼を証明に変える。1つはピッチ。プロトタイプ・仕様・ノートを1枚に束ね、先頭にデモ GIF を置く。人は動くところを見せられると承認が速い。相手は2種類いる。あなたと同じ不明点から出発するレビュアと、失敗点を潰したか確かめたい専門家だ。もう1つはクイズ。変更について自分に問題を出し、全問に正しく答えられるまで merge しない。答えられない問いは、まだ diff に残った不明点だ。エージェントが書いた変更なら、人間が受験する。diff を眺めるだけでは浅い。挙動は既存のコードパスに依存するし、長いセッションの後は自分が思うより先に進んでいる。

05指示は、細すぎても粗すぎても外れる

これは覚えておく価値がある。指示を細かく書きすぎると、方向転換すべき場面でも書いた手順に固執する。ざっくりしすぎると、あなたの案件に合わない「業界の定石」で勝手に埋める。狙うのは中間、意図は伝えつつ適応の余地を残す線だ。そして繰り返すが、結果が間違っていたら、まず疑うのは語尾ではなく unknowns の定義のほうだ。

06これを、スキルにした

ここまでの手順は、毎回手で思い出すには多い。だから finding-unknowns プラグインにまとめ、Claude Code が場面に応じて自分で引っ張れるようにした。導入は2行。

# Claude Code のプロンプトで
/plugin marketplace add SHU-T0/claude-tips
/plugin install finding-unknowns@claude-tips

中身は、前・最中・後にそのまま対応する。

ひと通り回すと、こうなる

「請求テーブル用の CSV インポータを作って」を、このプラグインで回した場合を追ってみる。

/finding-unknowns:unknowns 請求テーブル用の CSV インポータ

1. map-unknowns が起動し、まず「あなたはこのコードベースと CSV 取り込みにどれくらい慣れているか」を聞く。ここを答えるかどうかで、後の質問の精度が変わる。
2. 既知を確定する（請求テーブルのスキーマ、既存のバリデーション規約）。次に、アーキテクチャを変える質問から1問ずつ。「重複行はスキップか上書きか」「文字コードは UTF-8 固定でよいか」。各質問に推奨案が付くので、一言で返せる。
3. 「良い CSV インポータ」を自分が判定できないと気づいたら、learn-the-domain に落ちる。ストリーミングパースやエラー行の扱いの定石を教わってから戻る。
4. blind-spot-scout が走り、誰も言っていない前提を洗う。「アーカイブ済み行に API が null を返す」「10万行だと一括 INSERT がタイムアウトする」。
5. 揮発順（データモデル→型→UX フロー）に並べた計画と unknowns-map.md が手渡される。ここでは実装しない。
6. 実装は新しいセッションで。map と計画とプロトタイプを持ち込み、汚れていないコンテキストで書く。逸脱したら implementation-notes.md に3行、迷ったら保守的な方を選んで進む。
7. マージ前に verify-understanding。デモ GIF を先頭に、変更点と危ない箇所を1枚のピッチに束ねる。変更を書いていない側がクイズに答え、全問通るまで merge しない。

各スキルは単体でも使える。曖昧なタスクに着手する、判定基準がないまま案を並べそうになる、変更をマージしようとする。そのたびに、対応するスキルが description のトリガーで自分から立ち上がる。

07まとめ

賢いモデルを前にした仕事は、「手を動かす」から「不明点を定義し、検証を設計し、正しい場所に指示を置く」へ移る。次に何か作るときは、まず Claude に自分の不明点探しを手伝わせる。実装は、霧が晴れてからでいい。
